# 开户语音转发服务方案设计

## 1. 需求概述

当前项目需要提供统一的语音转发服务，对外输出稳定的 TTS 生成能力，并提供后台缓存管理能力。系统以 Spring Boot 为基础，围绕 TTS 合成、文本预处理、缓存复用、NER 识别、OSS 上传和后台缓存运维展开。

本方案以当前仓库实际实现为准，所有敏感信息均已脱敏为 `xxx`。

## 2. 建设目标

### 2.1 功能目标

- 提供批量文本转语音接口：`POST /api/tts/v1/generate`
- 提供后台缓存删除接口：`POST /admin/cache/v1/delete`
- 提供后台缓存查询接口：`GET /admin/cache/v1/value`
- 提供基于 `broker_id` 的模型路由能力
- 提供文本缓存、模板缓存和用户级 `traceId` 复用能力
- 提供音频异步上传对象存储的后置能力

### 2.2 非功能目标

- 统一日志链路，所有请求绑定 `traceId`
- 通过缓存降低重复文本的下游调用次数
- 通过模板匹配和 NER 降低姓名类文本处理成本
- 通过主备路由提升下游 TTS 可用性

## 3. 系统边界

当前版本包含：

- TTS HTTP 接口能力
- 后台缓存管理能力
- Redis 缓存能力
- NER 能力接入
- OSS/S3 对象存储接入

当前版本不包含：

- ASR 能力
- 数据库持久化
- Nacos 等配置中心依赖

## 4. 核心流程

### 4.1 TTS 主流程

1. Controller 接收请求并校验基础参数。
2. 服务端按 `user_id` 获取或生成 `traceId`。
3. 若开启鉴权，则校验 `sign` 和 `timestamp`。
4. 根据 `broker_id` 解析模型路由和主备端点。
5. 对文本做缓存判断、模板匹配和 NER 处理，生成 `final_text`。
6. 调用对应 TTS 下游完成音频合成。
7. 将可缓存结果写入 Redis。
8. 异步上传对象存储并回填 URL。
9. 返回 `audio_list`。

### 4.2 后台缓存流程

- 删除接口：校验 `admin-token` 后批量删除 key，并返回成功数量与失败列表。
- 查询接口：校验 `admin-token` 后读取指定 key 的当前值，并按 JSON 结构返回。

## 5. 分层设计

### 5.1 接口层

- `TtsGenerateController`
  - 路径：`/api/tts/v1/generate`
  - 职责：接收请求、绑定 `traceId`、调用 `TtsManager`
- `AdminCacheController`
  - 路径：`/admin/cache/v1/delete`
  - 路径：`/admin/cache/v1/value`
  - 职责：接收后台缓存管理请求并调用 `CacheAdminManager`

### 5.2 应用层

- `TtsManager`
  - 负责 TTS 请求编排
- `CacheAdminManager`
  - 负责后台缓存删除和查询编排

### 5.3 领域与基础设施层

- Redis：
  - 保存文本音频缓存
  - 保存缓存索引
  - 保存用户级 `traceId`
  - 保存 NER 模板缓存
- Feign：
  - 调用小模型 TTS
  - 调用大模型 TTS
  - 调用 NER
- OSS/S3：
  - 上传新生成音频并生成访问 URL

## 6. 缓存设计

### 6.1 缓存类型

| 缓存类型 | key 示例 | 作用 |
| --- | --- | --- |
| 音频 Base64 缓存 | `tts:cache:base64:{brokerId}:{modelCode}:{textMd5}` | 复用纯文本音频结果 |
| 音频 URL 缓存 | `tts:cache:url:{brokerId}:{modelCode}:{textMd5}` | 保存对象存储 URL |
| 音频索引缓存 | `tts:cache:base64:index:{brokerId}` | 支持按券商维度做容量控制 |
| 用户 traceId 缓存 | `speech:trace:user:{userId}` | 复用用户维度链路标识 |
| NER 模板缓存 | `tts:ner:template:{brokerId}` | 保存成熟模板与命中次数 |

### 6.2 关键规则

- `broker_id` 为空时统一归一到 `unknown`
- 纯文本音频缓存仅针对不包含 `nameTag` 的结果
- 命中缓存后会刷新索引访问时间
- 音频缓存按券商维度做数量上限控制
- NER 模板达到阈值后可直接参与匹配

## 7. 路由与降级设计

### 7.1 路由规则

配置中维护：

- `default-model-code`
- `broker-model-code-map`
- `primary-endpoint-map`
- `fallback-endpoint-map`

当前默认示例：

| broker | modelCode | 主端点 | 备端点 |
| --- | --- | --- | --- |
| `broker_a` | `hithink` | `SMALL` | `LARGE` |
| `broker_b` | `qwen` | `LARGE` | `SMALL` |
| 默认 | `hithink` | `SMALL` | `LARGE` |

### 7.2 降级规则

- 主端点整批失败后，整批切换到备端点重试一次
- NER 失败时降级为直接使用原文
- OSS 上传失败时不影响同步接口返回，只记录日志

## 8. 文本处理设计

### 8.1 原文已带标签

- 若原文已包含 `nameTag`，直接作为 `final_text`
- 不再走纯文本缓存复用

### 8.2 原文未带标签

1. 先按原文查询纯文本缓存
2. 未命中时先尝试模板匹配
3. 模板未命中再调用 NER 识别人名
4. 识别到人名后补齐 `<say-as interpret-as="name">`
5. 如有需要补齐 `<speak>` 标签

## 11. 风险与处理

| 风险点 | 影响 | 应对方式 |
| --- | --- | --- |
| 下游 TTS 不稳定 | 影响整批合成 | 主备端点切换，一次备轮重试 |
| NER 失败 | 姓名读法可能不准 | 降级使用原文，不阻断主链路 |
| 缓存膨胀 | Redis 占用增加 | 按券商维度做 TTL 和容量控制 |
| OSS 上传失败 | URL 为空 | 不影响同步返回，异步记录日志 |
| 敏感配置泄漏 | 安全风险 | 所有配置文档统一脱敏为 `xxx` |
